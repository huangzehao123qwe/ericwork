Context & Challenge

80% of admin time lost to manual compliance work

Procurement admins were executing compliance entirely by hand — manually reviewing every purchase request, writing internal guidelines from scratch, and personally conducting user training. The existing system technically worked, but it fundamentally lacked the empathy and scale needed for real enterprise use.

The Core Problem

The existing solution was technically functional but unusable at scale — it lacked contextual understanding of how procurement admins actually build compliance, and its rule management had no connection to the organizational structures admins already work within.

Research Insights

What admins actually needed

  • 🔁
    Admins struggled with inconsistent rule setup and were overwhelmed by the volume of manual, repetitive compliance work.
  • ⚙️
    The solution needed to automate repetitive manual work while keeping onboarding simple enough for non-technical admins.
  • 📈
    Scalability was a hard requirement — the design had to grow with the organization without creating performance bottlenecks.
  • 🛠
    Engineering complexity had to stay low — a maintainable architecture was as important as the user experience.
Solution Exploration

Three architectural approaches evaluated

We explored three distinct models for how compliance rules could be structured. Each had real trade-offs in precision, adoption, and scalability.

Approach Precision Scalability Adoption Performance
Item-Based
Rules per item		 High Low Low Issues
Item-Rule Based
Rules per item + category/vendor		 High Partial Moderate Moderate
User Group-BasedSelected
Rules per user group		 Highest Scalable High High

Why User Group-Based? It leverages existing org data — user hierarchy, cost centres, reporting lines — that admins already understand. This dramatically lowers the learning curve, improves precision, and aligns naturally with how organizations are structured.

Design Transformation

From aimless rule-linking to guided, targeted configuration

Before

Non-Guided Rule Building

  • No clear starting point or progression
  • Rules disconnected from user groups
  • Aimless linking between items and rules
  • No contextual guidance or scaffolding
After

Guided, Contextual Configuration

  • ✓ Step-by-step guided rule-building experience
  • ✓ Default + Exception context model (ordered priority)
  • ✓ Rules aligned to user groups & org structure
  • ✓ Targeted configuration with clear scoping

Key Highlights

What made this project stand out

01

Solving conflicting priorities through org-structure alignment

By anchoring rules to existing user groups — rather than items or categories — we resolved the tension between precision and scalability in a single architectural decision.

02

The right IA model had an outsized impact on adoption

Switching from item-based to user-group-based rules transformed onboarding from a steep cliff into a guided walk — dramatically increasing admin confidence and adoption speed.

03

Context isn't a field on a form — it's the frame that makes rules meaningful

Visualizing how rules apply to specific user groups in real time was the single most impactful design decision for building admin trust in the system.

Reflection

"User context isn't just another table of data — it's part of the experience!"

Next steps: The roadmap includes ML-driven rule suggestions that learn from existing org patterns, and a Live Preview mode so admins can see the impact of rule changes on user groups before publishing.
