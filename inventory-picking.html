01

The problem

A consumer-style app used by professionals all shift, every day

The problem

A phone app built for casual users, used by pros

These screens are the current warehouse picking flow—pick a site, open your list, confirm each item. The layout is perfectly fine if you touch the app now and then. Our users live in it all shift, and the patterns underneath it were shared with consumer-style mobile apps: airy rows, soft generous padding, one action per screen. Every extra tap multiplies across thousands of picks a day.

02

The decision

When tightening the flow isn't enough

First attempt

One item per screen — cleaner, still too spacey

The first direction kept the existing mobile components and just simplified the flow: one pick at a time, scanner up top, big quantity keypad. It helped, but we were still tightening steps inside a design system built for casual users. The screen itself stayed airy. The team agreed: to really move the needle for pros, we had to change the building blocks, not just the flow.

03

The design system

A new component layer built for pro users

The real fix

Building a mobile design system for pro users

Workday already had two component libraries I could stand on: a mature web component library and an older mobile kit built around casual, consumer-style phone use. I used both as a starting point—same colors, same rules, same accessibility baseline—and designed a new pro-user layer on top: tighter action bars, compact item rows, inline form fields, a first-class scanner input, and a multi-value container so one screen can hold a whole batch. Denser, but still familiar.

The preview below is that library running live. Tap Pattern library to browse every component I built, or jump into a real task to see them composed together. It’s the same React code that ships—not a deck.

04

The final design

Batch pick with the new library — denser, scanner-led, webview-ready

The final design

Batch pick with the new library — denser, scanner-led

With the pro-user components in place, batch pick holds many more lines in one thumb’s reach without feeling cramped. On top of that, I added two task-specific touches: the barcode scanner drives the next step on its own, and required fields auto-advance—type quantity, move to the tote field, scan, move on—so your thumb never hunts for the next box. This runs in a webview, so engineering could ship it without a full native rebuild.

05

Designer in the stack

Extending the design system and shipping it

Designer in the stack

Extending the design system — and shipping it

Workday's design system was built for web and native mobile. Mobile webviews — where these inventory tasks run — had no formal component layer. Working directly with engineering, I filled that gap: denser proportions, scanner-aware patterns, and a component registry both designers and engineers could reference.

I was among the first designers at Workday to push components into production. Using Cursor for frontend iteration, Lovable for rapid prototyping, and Figma MCP to keep design tokens in sync with the Workday design system, I compressed the handoff into a workflow I owned end-to-end. That practical fluency let me make faster, higher-fidelity decisions with engineering — and cut the back-and-forth that typically slows mobile webview work.

The library didn't stay in one team. Other groups building mobile webview experiences picked it up as a foundation — turning a task-specific component set into a reusable extension of the core system, and reducing the time other teams spent re-solving the same density and interaction problems from scratch.

Outcomes at Workday

What shipped

  • Pro-user mobile component library shipped — built on Workday's web and mobile base, same tokens, denser proportions for shift-long use
  • Batch pick redesigned with scanner-driven control and auto-advancing fields — measurably fewer taps per pick cycle
  • Library extended to 21 inventory tasks — one source of truth shared with engineering, adopted by other mobile webview teams
  • First designer at Workday to push components to production using Cursor, Lovable, and Figma MCP — collapsing design-to-code handoff into a single workflow
Takeaways

What I carry into the next project

01

Design for who actually uses it

The same mobile patterns can be right for casual users and wrong for pros. When the flow can’t get denser, the system is the thing to redesign.

02

Build on what’s already there

The new pro-user library stood on Workday’s existing web and mobile libraries. Same tokens, same rules—just new proportions.

03

Small task touches go a long way

Scanner-driven control and auto-field advancement aren’t components—they’re behaviors. The library made room for them without bending the rules.

04

Tools that close the design–engineering gap

Cursor, Lovable, and Figma MCP let me own both the spec and the implementation. Faster decisions, fewer handoff gaps, and a library engineering actually trusts.
